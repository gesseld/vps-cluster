## ⚠️ Implementation Sequence: Non-Negotiable Order

> **Short answer: Yes — the resource budget scaffolding must be implemented *before* deploying individual plane components.**

This isn't about writing application code first; it's about establishing Kubernetes-level guardrails that enforce your budget constraints at scheduling time. Without these in place, pods can schedule without limits, leading to resource contention, eviction storms, or silent OOM kills.


## 1. Architectural Foundation

### 1.1 Design Principles (Sequence-Aware)

| Principle | Implementation | Rationale | Sequence Dependency |
|-----------|----------------|-----------|-------------------|
| **Zero-Trust Boundaries** | mTLS between all planes; SPIFFE identities for workload attestation; NetworkPolicy default-deny with explicit allow + egress restrictions | Prevent lateral movement; enforce least-privilege network access | NetworkPolicy CRD + default-deny must exist before any workload deployment |
| **Resource Frugality** | 8.5GB RAM realistic budget for foundation (53% of 16GB cluster); aggressive requests=limits to prevent overcommit; tiered PriorityClasses | Account for real-world overhead; prevent noisy-neighbor eviction | **PriorityClasses + ResourceQuotas MUST be applied before any pod spec references them** |
| **High Availability** | 3-replica NATS JetStream cluster; PostgreSQL primary + async read replica; Temporal 2-replica active-active; topology spread constraints with `requiredDuringScheduling` | Eliminate single points of failure for critical path services | Node labels + StorageClass with `WaitForFirstConsumer` must exist before stateful workloads schedule |
| **Unified Telemetry** | Fluent Bit (not Promtail+OTel) for logs + metrics shipping; VictoriaMetrics for TSDB with cardinality controls; 2s end-to-end latency target | Reduce resource footprint while maintaining observability fidelity | Observability plane deployed last to avoid self-monitoring bootstrap issues |
| **Immutable Infrastructure** | GitOps via ArgoCD; Kyverno policies enforce labels/limits; all changes tracked in Git; schema migrations via Atlas/Flyway | Ensure reproducibility, auditability, and safe schema evolution | ArgoCD deployed after Kyverno to enforce policy-on-deployment |
| **Backpressure-Aware** | Queue depth monitoring on NATS; circuit breaker patterns in execution plane; API server QoS controls | Prevent cascade failures during load spikes | Monitoring must be active before load-testing execution plane |

### 1.2 Resource Budget (Foundation Planes - Realistic)

| Plane | RAM Request | RAM Limit | CPU Request | CPU Limit | Critical Pods | Notes |
|-------|-------------|-----------|-------------|-----------|---------------|-------|
| **Control** | 2.8GB | 4.2GB | 1.8 | 3.2 | Temporal×2 (750MB ea), Kyverno, SPIRE | +700MB for Temporal HA |
| **Data** | 4.2GB | 7.1GB | 2.8 | 4.8 | PostgreSQL primary+replica, NATS×3 (170MB ea), MinIO, Redis | +800MB for read replica |
| **Observability** | 1.5GB | 2.9GB | 1.0 | 1.9 | Fluent Bit (80MB/node), VMSingle (512MB+snapshot), Loki | +300MB for backup overhead |
| **Total** | **8.5GB** | **14.2GB** | **5.6** | **9.9** | — | Realistic baseline |
| **Headroom** | **7.5GB** | **1.8GB** | **6.4** | **2.1** | For Execution + AI planes | Tight but workable |

> **Burst Strategy & Priority Tiers:**
> - K3s allows 2× request overcommit. Foundation pods use tiered `PriorityClass` to prevent eviction during resource pressure.
> - `foundation-critical`: PostgreSQL, NATS, Temporal (priority 1000000)
> - `foundation-high`: Kyverno, SPIRE, MinIO (priority 900000)
> - `foundation-medium`: Observability components (priority 800000)
> - All foundation pods set `preemptionPolicy: PreemptLowerPriority`
> - **⚠️ These PriorityClasses must be created BEFORE any pod spec references them**

## 2. Phase 0: Budget Scaffolding (MANDATORY FIRST STEP)

### 2.1 Purpose
Encode resource constraints as Kubernetes primitives that the scheduler enforces at admission time. This phase creates the guardrails that make "resource frugality" enforceable, not aspirational.

### 2.2 Implementation Tasks

#### **Task BS-1: PriorityClasses Deployment**
**Objective:** Create tiered priority classes that protect critical foundation workloads during resource pressure.

**Sub-tasks:**
- Apply `foundation-critical` (value: 1000000) for PostgreSQL, NATS, Temporal
- Apply `foundation-high` (value: 900000) for Kyverno, SPIRE, MinIO
- Apply `foundation-medium` (value: 800000) for observability components
- Set `preemptionPolicy: PreemptLowerPriority` on all classes
- Document priority hierarchy in `shared/priority-classes.md`

**Deliverables:**
```yaml
# shared/priority-classes.yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: foundation-critical
value: 1000000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: "Critical foundation: PostgreSQL, NATS, Temporal"
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: foundation-high
value: 900000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: "High-priority foundation: Kyverno, SPIRE, MinIO"
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: foundation-medium
value: 800000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: "Medium-priority foundation: Observability components"
```

**Validation:**
```bash
kubectl get priorityclass | grep foundation
# Expected: foundation-critical, foundation-high, foundation-medium all present
```

#### **Task BS-2: ResourceQuotas + LimitRanges per Namespace**
**Objective:** Enforce namespace-level resource budgets and container-level defaults.

**Sub-tasks:**
- Create `ResourceQuota` for each foundation namespace (control-plane, data-plane, observability-plane)
- Set hard limits matching budget table (e.g., control-plane: 2.8Gi request memory)
- Create `LimitRange` to inject default requests/limits for containers without explicit specs
- Set `max` limits to prevent runaway pods
- Document quota rationale in `shared/resource-budget.md`

**Deliverables:**
```yaml
# shared/resource-quotas.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: control-plane-quota
  namespace: control-plane
spec:
  hard:
    requests.memory: "2.8Gi"
    limits.memory: "4.2Gi"
    requests.cpu: "1.8"
    limits.cpu: "3.2"
    pods: "15"
---
apiVersion: v1
kind: LimitRange
metadata:
  name: control-plane-defaults
  namespace: control-plane
spec:
  limits:
  - type: Container
    defaultRequest:
      memory: "256Mi"
      cpu: "100m"
    default:
      memory: "512Mi"
      cpu: "250m"
    max:
      memory: "1Gi"
      cpu: "1"
```

**Validation:**
```bash
kubectl describe resourcequota -n control-plane
kubectl describe limitrange -n control-plane
# Verify hard limits match budget; defaults are reasonable
```

#### **Task BS-3: StorageClass with WaitForFirstConsumer**
**Objective:** Ensure stateful workloads schedule to nodes with available NVMe before PVC binding.

**Sub-tasks:**
- Define `nvme-waitfirst` StorageClass with `volumeBindingMode: WaitForFirstConsumer`
- Set `allowVolumeExpansion: true` for future growth
- Document CSI driver compatibility (Rancher CSI, Longhorn, etc.)
- Test PVC creation with topology constraints

**Deliverables:**
```yaml
# shared/storage-classes.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nvme-waitfirst
provisioner: rke.csi.rancher.io  # Adjust for your CSI driver
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
reclaimPolicy: Retain
```

**Validation:**
```bash
kubectl get storageclass nvme-waitfirst -o jsonpath='{.volumeBindingMode}'
# Expected: WaitForFirstConsumer
```

#### **Task BS-4: Node Labeling for Topology Awareness**
**Objective:** Enable topology-aware scheduling for I/O-heavy workloads.

**Sub-tasks:**
- Label 2 of 3 nodes as `node-role=storage-heavy` for PostgreSQL + MinIO placement
- Leave 1 node general-purpose for control/observability workloads
- Document labeling strategy in `shared/topology.md`
- Create script for reproducible labeling

**Deliverables:**
```bash
#!/bin/bash
# scripts/label-nodes.sh
set -e
NODES=$(kubectl get nodes -o name | cut -d/ -f2)
# Label first two nodes as storage-heavy
echo "$NODES" | head -n 2 | while read node; do
  kubectl label node "$node" node-role=storage-heavy --overwrite
done
echo "✅ Nodes labeled: $(kubectl get nodes -l node-role=storage-heavy -o name | wc -l) storage-heavy"
```

**Validation:**
```bash
kubectl get nodes -l node-role=storage-heavy
# Expected: 2 nodes labeled
```

#### **Task BS-5: NetworkPolicy CRD + Default-Deny Template**
**Objective:** Ensure network isolation primitives are available before workloads deploy.

**Sub-tasks:**
- Verify NetworkPolicy CRD is installed (Cilium/CNI requirement)
- Create default-deny template for ingress + egress
- Document policy pattern for plane-specific policies
- Test policy application on dummy pod

**Deliverables:**
```yaml
# shared/network-policy-template.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: {{ .Namespace }}
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
  # No rules = deny all by default
```

**Validation:**
```bash
kubectl api-resources | grep networkpolicies
# Expected: networkpolicies present
kubectl apply -f shared/network-policy-template.yaml --dry-run=client
# Expected: no errors
```

## 4. Phase 2: Data Plane Implementation (Stateful Dependencies First)

> **Why Data Plane before Control Plane?** Temporal (now in Data Plane) depends on PostgreSQL for persistence. Deploy dependencies before dependents. Control Plane components (Kyverno, SPIRE, ArgoCD) depend on shared foundations.

### 4.1 Purpose & Scope
The **memory and nervous system**: persistent storage, event streaming, caching, and object storage. I/O bound, optimized for throughput. Contains the only stateful data in the foundation.

### 4.2 Implementation Tasks

#### **Task DP-1: PostgreSQL with RLS, Pooling & Read Replica**
**Objective:** Tenant-isolated database with connection pooling and read scaling.

**Sub-tasks:**
- Deploy PostgreSQL 15 primary (StatefulSet, 50GB PVC) with `volumeBindingMode: WaitForFirstConsumer`
- Deploy PostgreSQL 15 **async read replica** (StatefulSet, 50GB PVC) on separate node
- Enable `pgcrypto` extension for UUID generation
- Configure Row-Level Security (RLS):
  - `documents` table: `USING (tenant_id = current_setting('app.current_tenant')::UUID)`
  - `workflows` table: RLS by Temporal namespace
- Deploy pgBouncer (transaction pooling mode):
  - `max_client_conn = 500`
  - `default_pool_size = 20`
  - Route writes to primary, reads to replica via `app.read_only` GUC
- Create databases: `app`, `temporal_visibility`, `spire`
- Automated backups: Daily base backup + WAL archiving to MinIO bucket `backups/postgres`
- **Schema migrations**: Integrate Atlas/Flyway init container for versioned migrations
- **Topology spread**: PostgreSQL primary and replica anti-affinity with MinIO (don't colocate I/O heavy pods); label nodes `node-role=storage-heavy`

**Deliverables:**
- `data-plane/postgresql/primary-statefulset.yaml`
- `data-plane/postgresql/replica-statefulset.yaml`
- `data-plane/postgresql/pgbouncer.yaml` (with read/write routing)
- `data-plane/postgresql/init-scripts/01-rls.sql`
- `data-plane/postgresql/init-scripts/02-tenants.sql`
- `data-plane/postgresql/migrations/atlas-config.yaml` (or flyway)
- `data-plane/postgresql/backup-cronjob.yaml`
- Secret `postgres-superuser`, `postgres-app-user`

**Validation:**
```bash
# RLS test
psql -c "SET app.current_tenant = 'A'; SELECT * FROM documents;" # returns only tenant A
# Read replica routing
psql -c "EXPLAIN ANALYZE SELECT * FROM documents WHERE id=1;" # shows replica host
```

#### **Task DP-2: NATS JetStream HA Cluster (3-Replica + Backpressure)**
**Objective:** High-availability event bus with persistence and backpressure controls.

**Sub-tasks:**
- Deploy NATS Server 2.10+ with JetStream (3 replicas, Parallel pod management)
- **Resource per replica:** 170MB request / 256MB limit
- Configure cluster routes for mesh (3-node cluster)
- Define Streams with backpressure monitoring:
  - `DOCUMENTS`: Subjects `data.doc.>`, retention=WorkQueue, max-msgs=100k, replicas=3, `max_bytes: 5GB`
  - `EXECUTION`: Subjects `exec.task.>`, retention=Interest, max-age=24h, replicas=3
  - `OBSERVABILITY`: Subjects `obs.metric.>`, retention=Limits, max-bytes=1GB, replicas=2
- Enable TLS for client connections (port 4222)
- **Topology spread**: One replica per node mandatory (maxSkew: 0, `whenUnsatisfiable: DoNotSchedule`)
- **Backpressure monitoring**: Export `nats_jetstream_stream_consumer_pending_msgs` to vmagent; alert on queue depth >80%
- Add **PodDisruptionBudget**: `minAvailable: 2` (maintain quorum)

**Deliverables:**
- `data-plane/nats/values.yaml` (Helm values with 3 replicas)
- `data-plane/nats/stream-config.yaml` (with backpressure limits)
- `data-plane/nats/networkpolicy.yaml` (allow from execution, control, observability)
- `data-plane/nats/pdb.yaml`
- `data-plane/nats/metrics-exporter.yaml`

**Validation:**
```bash

# Metrics show consumer pending <80% threshold
```

## 5. Phase 3: Control Plane Implementation (Orchestration Layer)

> **Why Control Plane after Data Plane?** Kyverno requires SPIRE; ArgoCD requires Git repo access. Control Plane components manage orchestration and policy enforcement.

### 5.1 Purpose & Scope
The **governance layer**: policy enforcement, identity management, and GitOps. Manages security, compliance, and deployment orchestration across all planes.

### 5.2 Implementation Tasks



#### **Task CP-1: Kyverno Policy Engine (Consolidated + Rate Limiting)**
**Objective:** Replace OPA with Kyverno for lower overhead and native Kubernetes UX, with API protection.

**Sub-tasks:**

## 8. Validation & Testing Framework

### 8.1 Pre-Phase Validation Scripts (Enhanced)

```bash
#!/bin/bash
# scripts/validate-phase-gates.sh

set -e

validate_gate_0() {
  echo "🔒 Validating Phase 0: Budget Scaffolding..."
  kubectl get priorityclass foundation-critical foundation-high foundation-medium >/dev/null 2>&1 || { echo "❌ PriorityClasses missing"; exit 1; }
  kubectl get resourcequota -n control-plane data-plane observability-plane >/dev/null 2>&1 || { echo "❌ ResourceQuotas missing"; exit 1; }
  kubectl get storageclass nvme-waitfirst -o jsonpath='{.volumeBindingMode}' | grep -q WaitForFirstConsumer || { echo "❌ StorageClass misconfigured"; exit 1; }
  [ $(kubectl get nodes -l node-role=storage-heavy --no-headers | wc -l) -ge 2 ] || { echo "❌ Insufficient storage-heavy nodes"; exit 1; }
  echo "✅ Gate 0 passed"
}

validate_gate_1() {
  echo "🔒 Validating Phase 1: Shared Foundations..."
  kubectl get pods -n control-plane -l app=spire-server -o jsonpath='{.items[0].status.phase}' | grep -q Running || { echo "❌ SPIRE not ready"; exit 1; }
  kubectl auth can-i create pod --as=system:serviceaccount:default:default --namespace=control-plane | grep -q no || { echo "❌ RBAC not restrictive"; exit 1; }
  echo "✅ Gate 1 passed"
}

# Add validate_gate_2, validate_gate_3, validate_gate_4 similarly

case "${1:-all}" in
  0) validate_gate_0 ;;
  1) validate_gate_1 ;;
  2) validate_gate_2 ;;
  3) validate_gate_3 ;;
  4) validate_gate_4 ;;
  all) validate_gate_0 && validate_gate_1 && validate_gate_2 && validate_gate_3 && validate_gate_4 ;;
  *) echo "Usage: $0 {0|1|2|3|4|all}"; exit 1 ;;
esac
```

## 10. Success Gates & Exit Criteria (Sequence-Enforced)

The Foundation Planes are considered **Production Ready** when:

| Gate | Phase | Criteria | Sign-off | Validation Method |
|------|-------|----------|----------|-------------------|
| **G0** | Phase 0 | All scaffolding resources applied + validated | Automated | `scripts/validate-phase-gates.sh 0` |
| **G1** | Phase 1 | PKI + RBAC + NetworkPolicy functional | Automated + Manual | SPIRE SVID test + RBAC audit + network isolation test |
| **G2** | Phase 2 | Data Plane HA + topology + resource compliance | Manual (chaos) | Kill PostgreSQL primary; verify replica promotion |
| **G3** | Phase 3 | Control Plane stable + policy-enforced + GitOps synced | Automated + Manual | ArgoCD sync test + Kyverno policy block test |
| **G4** | Phase 4 | Observability collecting + alerting + cardinality controlled | Automated | Log latency test + series count query + alert firing test |
| **G5** | Cross-Phase | End-to-end workflow: doc ingest → Temporal → storage → metrics | Manual | Submit test document; verify full pipeline telemetry |

**Exit Criteria:** All gates G0-G5 passed with documented evidence in `validation-reports/`. **Gate 0 is mandatory before any other gate can be attempted.**

## 11. Operational Runbooks (Sequence-Aware)

### 11.1 Emergency Procedures

| Scenario | Sequence-Aware Response |
|----------|------------------------|
| **Temporal outage** | 1) Verify Phase 0 scaffolding intact (PriorityClasses) 2) Check Data Plane dependencies (PostgreSQL) 3) Scale Temporal replicas |
| **NATS quorum loss** | 1) Confirm ResourceQuotas not causing eviction 2) Verify topology spread constraints still satisfied 3) Restore 3rd replica |
| **PostgreSQL primary failure** | 1) Ensure `storage-heavy` labels still correct 2) Promote replica 3) Update connection strings 4) Verify Phase 0 StorageClass still `WaitForFirstConsumer` |
| **Kyverno blocking legitimate pods** | 1) Check Phase 1 RBAC baseline 2) Review LimitRange defaults 3) Temporarily relax policy via GitOps PR (not kubectl edit) |
| **Observability silent failure** | 1) Verify Phase 4 deployed after Phase 3 2) Check NetworkPolicy egress rules 3) Validate vmagent scrape targets |
| **SPIRE failure** | Toggle fallback mode to cert-manager; monitor SVID issuance; restore SPIRE |
| **MinIO replication lag** | Check network; verify `mc mirror --watch` process; increase resources if needed |

### 11.2 Scaling Guidelines

```yaml
# When adding Execution/AI planes:
# 1. FIRST: Re-run Phase 0 validation to ensure headroom
scripts/validate-phase-gates.sh 0

# 2. THEN: Update ResourceQuotas if needed (GitOps PR)
# 3. THEN: Deploy new plane with appropriate PriorityClass
# 4. FINALLY: Update observability dashboards to include new plane
```

**Scaling Rules:**
- **Add Execution/AI planes**: Ensure foundation headroom >20%; monitor resource budget dashboard
- **Increase document volume**: Scale NATS consumers first; then PostgreSQL read replicas; then Temporal workers
- **Add nodes**: Label new nodes; update topology spread constraints; rebalance stateful workloads

### 11.3 Upgrade Procedures

| Component | Sequence Consideration |
|-----------|----------------------|
| **K3s upgrade** | Drain nodes respecting PriorityClasses; foundation-critical pods evicted last |
| **PostgreSQL upgrade** | Upgrade replica first; ensure StorageClass still `WaitForFirstConsumer` for new PVCs |
| **Kyverno policy update** | Test in staging with Phase 0 ResourceQuotas; use GitOps PR workflow |
| **Observability component upgrade** | Upgrade vmagent before VictoriaMetrics; ensure cardinality limits still enforced |
| **Schema migrations** | Run Atlas/Flyway in dry-run mode first; backup before applying; monitor application logs |

## Appendix A: "What If I Skip the Sequence?" - Risk Matrix

| Skipped Phase | Immediate Consequence | Latent Risk | Recovery Complexity |
|--------------|----------------------|-------------|-------------------|
| **Phase 0: Budget Scaffolding** | Pods schedule without limits; resource contention within hours | Silent OOM kills; cascade failures during load spikes | High: Must drain nodes, re-apply quotas, restart workloads |
| **Phase 1: Shared Foundations** | mTLS failures; unauthorized cross-plane traffic | Security boundary breach; audit gap | Medium: Re-deploy PKI; rotate credentials; audit access logs |
| **Phase 2 before Phase 3** | Temporal crashes waiting for PostgreSQL; workflow loss | Data corruption if partial writes occur | Medium: Restore from backup; replay workflows |
| **Phase 4 before Phase 3** | Observability agents scrape non-existent targets; metric cardinality explosion | Alert fatigue; missed critical signals | Low: Reconfigure scrapes; reset cardinality limits |

## Appendix B: Iterative Development Tip

You don't need *perfect* budget numbers on day one. Start conservative and iterate:

```yaml
# Start with buffer, then tighten after load testing
resources:
  requests:
    memory: "750Mi"  # Start 25% above spec
  limits:
    memory: "1Gi"    # Allow burst headroom
```

Then use observability data to right-size:
```bash
# After 7 days of production load:
kubectl top pods -n data-plane --containers
# Adjust requests/limits to p95 usage + 20% buffer
# Update Phase 0 ResourceQuotas via GitOps PR
```

**Key**: Even with iterative tuning, Phase 0 scaffolding must exist *before* workloads deploy. You can adjust the numbers later via GitOps, but the guardrails must be in place from day one.
